## 1. Проблема

Сегодня боссы (`subject_bosses`) создаются в **трёх разных местах**, каждое со своими проблемами:

| Место | Триггер | Поведение | Проблема |
|---|---|---|---|
| `POST /api/admin/create-user` | создание учителя | Для всех классов школы × subjects учителя | Считает HP по текущему `studentCount`, а учеников часто добавляют ПОСЛЕ учителя → боссы получают HP=1000 (min) |
| `POST /api/bosses/ensure` | первый рендер teacher-UI | Лениво, per (class, subject) | Тот же баг: момент создания непредсказуем |
| `/teacher/boss/new` | ручное действие учителя | HP вводится слайдером | Не залинкован из навигации, не используется |

**Итог:** HP босса зависит от того, **в какой момент был триггер**, а не от реального размера класса на старте сезона. Для альфы это критично: 15 учеников × математика → по формуле 3600 HP, но если учитель зашёл до наполнения — получит 1000 HP → босс падёт за 1-2 дня при xp_multiplier=300%.

**Side-finding (из ре-проверки):** в `grade-batch/route.ts` (строки 302-340) и `action/route.ts` (строки 237-272) есть обращения к таблицам `season_boss`/`season_boss_class_hp`, которых **в БД не существует**. НО: `maybeSingle()` молча возвращает null, и fallback-логика корректно переключается на реальную таблицу `subject_bosses`. То есть **урон по боссу в production доходит до БД**, но код грязный (каждый graded quest делает два зря-запроса в несуществующие таблицы). Cleanup рекомендуется, но не блокер альфы (см. §7).

---

## 2. Цель

Сделать создание боссов **детерминированным**: всё создание происходит в один момент (активация сезона) на основе **фактического размера класса на этот момент**. Админ получает кнопку для явного пересчёта, если состав класса изменился в процессе.

---

## 3. Новое поведение

### 3.1 При активации сезона
`POST /api/admin/activate-season` теперь:
1. Как и раньше: деактивирует другие активные сезоны школы, ставит текущему `status='active'`.
2. **Новое:** для каждого класса школы с `studentCount ≥ 1`:
   - Собирает список предметов: все уникальные subjects всех teachers школы, после `normalizeSubject` и дедупа (case-insensitive).
   - Для каждого предмета, если нет `subject_bosses` с `(season_id, class_id, subject_id)` — создаёт:
     - `max_hp = current_hp = calculateBossHp({ studentCount, seasonWeeks })`
     - `name = "Босс: {subject}"`, `avatar = '🐉'`, `is_defeated = false`
3. Возвращает `{ success: true, bossesCreated: N, classesSkipped: [...] }` для прозрачности.

**Edge-cases:**
- Класс с 0 учеников → пропускается (`classesSkipped` включает id+имя).
- В школе нет учителей с `subjects` → `bossesCreated: 0`, но активация успешна.
- Повторная активация того же сезона (после деактивации → активации) — **idempotent**: `subject_bosses` уникальны по `(season_id, class_id, LOWER(subject_id))` (миграция `subject_bosses_dedupe` уже есть). Дубликаты не плодим.

### 3.2 Ручной пересчёт HP
Новый endpoint `POST /api/admin/recalculate-boss-hp`:
- **Input:** `{ seasonId: string, dryRun?: boolean }`
- **Output:** `{ changes: BossChange[], newBosses: NewBoss[], skipped: SkippedBoss[] }` где
  - `BossChange = { bossId, className, subjectId, oldMaxHp, newMaxHp, oldCurrentHp, newCurrentHp }`
  - `NewBoss = { className, subjectId, maxHp }` — боссы, которых ещё не было (новый предмет/новый учитель)
  - `SkippedBoss = { bossId, reason }` — например, `reason: 'defeated'`
- **Поведение:**
  1. Загрузить все `subject_bosses` для `season_id`.
  2. Для каждого класса с `studentCount ≥ 1` пересчитать `calculateBossHp({ studentCount, seasonWeeks })`.
  3. Для каждого существующего босса:
     - Если `is_defeated=true` → **пропустить полностью** (не менять ни `max_hp`, ни `current_hp`).
     - Иначе:
       - `new_max_hp = calculated`
       - `new_current_hp = min(old_current_hp, new_max_hp)` (clamp — сохраняем нанесённый урон)
  4. Дополнительно: если среди subjects школы появился новый (которого не было на активации) → создать нового босса для каждого класса с ≥1 учеником.
  5. Если `dryRun=true` — **только вернуть diff**, ничего не писать. Если `false` — применить.

### 3.3 UI: кнопка в админке сезонов
В `src/app/(admin)/admin/seasons/page.tsx` для **активных** сезонов рядом с "Завершить сезон" добавить кнопку "**🔄 Пересчитать HP боссов**":
1. Клик → `POST /api/admin/recalculate-boss-hp` с `dryRun: true`.
2. Модалка подтверждения (или нативный `confirm`) с текстовым diff:
   ```
   Пересчёт HP для сезона "Майский Квест":
   
   Изменения:
   • 6А · Математика: 1000 → 3600 HP (текущий: 1000 → 1000)
   • 6Б · Математика: 1000 → 2400 HP (текущий: 850 → 850)
   
   Новые боссы: 1
   • 6А · Физика: 3600 HP
   
   Пропущены (повержен, награды розданы):
   • 6В · Математика
   
   Применить?
   ```
3. Подтверждение → повторный вызов с `dryRun: false`. Toast-feedback.

Стилистика кнопки — как у существующей "Завершить сезон", но с purple-акцентом (`rgba(139,92,246,...)`), иконка `🔄`.

## 4. Что удаляется / остаётся

| Компонент | Действие | Причина |
|---|---|---|
| `POST /api/admin/create-user` — блок создания боссов (строки 146-195) | **Удалить** | Дублирует новую логику активации, создавал боссов в неправильный момент |
| `POST /api/bosses/ensure` + `useSubjectBosses` hook | **Оставить** | Fallback: если учитель добавлен ПОСЛЕ активации, он получит боссов при первом заходе в UI |

| `boss_hp_multiplier` в `economy_config` | **Не трогать в этой задаче** | Dead field, но его удаление — отдельная задача (см. post-alpha) |

## 5. Технические детали

### 5.1 Source of truth для subjects при активации
```sql
SELECT DISTINCT unnest(subjects) as subject
FROM users
WHERE school_id = $1 AND role = 'teacher' AND subjects IS NOT NULL;
```
Затем `normalizeSubject()` на каждом + дедуп через `Set` по lowercase.

**Ограничение (accepted для альфы):** если в школе 10 учителей с 10 разными предметами, каждый класс получит 10 боссов — даже если по факту не все предметы ведутся в каждом классе. Для альфы в Циркуле (1 учитель, 1 предмет) это идеально.

**Post-alpha:** миграция `class_teachers(class_id, teacher_id, subject)` + UI назначения — отдельная задача.

### 5.2 Race conditions
- Активация сезона не ожидается параллельно с добавлением учеников. Если всё же → при пересчёте админ увидит корректные цифры.
- `/api/bosses/ensure` (fallback) параллельно с активацией: уникальный индекс `(season_id, class_id, LOWER(subject_id))` защищает от дубликатов, error code 23505 обрабатывается в ensure корректно (уже реализовано).

### 5.3 Транзакционность
Активация сезона должна оставаться атомарной хотя бы на уровне "статус сезона + боссы":
- Обернуть в Supabase RPC, или
- Сначала обновить статус, потом создавать боссов, при ошибке боссов — откатить статус.
- **Выбор:** простой последовательный код, при ошибке создания боссов — лог + возврат `{ warning: "..." }`, но статус не откатываем (активация важнее, боссов можно доcоздать через "Пересчитать HP"). Компромисс pragmatic, помечен в логах.

## 6. Тестирование

### Unit-тесты (vitest)
- `calculateBossHp()` — уже покрыт (`boss-hp.test.ts`), не трогаем.
- Новая функция `collectSchoolSubjects(schoolId)` — возвращает дедуплицированный список.
- Новая функция `buildRecalcPlan(seasonId, dryRun)` — возвращает diff без применения.

### Интеграционные тесты
- `activate-season` flow: создать школу → класс → учеников → учителя → активировать → проверить что `subject_bosses` создан с правильным HP.
- `recalculate-boss-hp` flow: активировать → добавить учеников → пересчитать → проверить что `max_hp` вырос, `current_hp` clamped, defeated не тронут.
- Активация без учеников → `classesSkipped` непустой, боссов не создано.
- Активация без учителей с subjects → `bossesCreated: 0`.

### Ручное тестирование для альфы
1. Создать школу Циркуль (уже есть).
2. Добавить учителя математика с `subjects=['Математика']`.
3. Добавить 15 учеников в класс "6".
4. Активировать сезон "Майский Квест".
5. Проверить в БД: `subject_bosses` для (season, class=6, subject='Математика') с `max_hp = 15 × 3 × 4 × 20 = 3600`.
6. Добавить ещё 5 учеников.
7. Нажать "Пересчитать HP" → увидеть diff 3600 → 4800, подтвердить.

## 7. Scope + out of scope

### В scope
- Изменения в `activate-season` route.
- Новый endpoint `recalculate-boss-hp`.
- UI-кнопка в seasons page + модалка diff.
- Удаление кода из `create-user`, удаление `/teacher/boss/new`.
- Unit + integration тесты.

### В scope (opportunistic cleanup)
- Убрать dead queries к `season_boss`/`season_boss_class_hp` в `grade-batch/route.ts:302-340` и `action/route.ts:237-272`. Оставить только запрос к `subject_bosses`. Функциональность не меняется (работает сейчас через fallback), но код чище и нет зря-запросов в несуществующие таблицы на каждую сданную домашку.

### Out of scope (post-alpha)
- Миграция `class_teachers` и UI назначения teacher↔class↔subject.
- Удаление dead fields `boss_hp_multiplier` / `hp_regen_rate` из `economy_config`.
- Переработка `alphaSimulation.ts` под реальную формулу (сейчас использует фиктивный HP=63000).
- Расширение slider-ов в admin UI (max=400) — dead fields всё равно.

## 8. Риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Grязные запросы к несуществующим таблицам в grade-batch/action | Низкая (работает через fallback) | Opportunistic cleanup в scope §7. Без cleanup тоже работает, просто -2 лишних запроса на сданную домашку. |
| При активации у пустых классов не создаются боссы, админ не замечает | Средняя | `classesSkipped` в ответе + toast с warning в UI |
| Учитель зашёл до активации сезона → `ensure` не создаст (нет active season) → после активации при следующем заходе создаст **с правильным HP** | Низкая | Поведение уже корректное благодаря `ensure` |
| Удаление блока из `create-user` ломает демо-пути (`/api/demo/setup`) | Низкая | Проверить в тестах, при необходимости обновить demo-setup |
| Двойная раздача наград при воскрешении defeated босса | Не применимо | Q3=a: defeated пропускаются полностью |
