# God Mode — Admin View-As-Student

**Дата:** 2026-04-09
**Автор:** Hero Academy team
**Статус:** Design approved, ready for implementation plan

## Цель

Дать администратору возможность открыть интерфейс любого ученика и увидеть игру «глазами его героя» — для отладки, поддержки и диагностики проблем учеников. Это режим просмотра, а не полного управления.

## Scope (MVP)

**В MVP входит:**
- Просмотр трёх ключевых экранов ученика: Герой, Инвентарь, Квесты
- Навигация между этими экранами и просмотр деталей (карточка артефакта, карточка квеста)
- Ручная кнопка «Обновить» + авто-обновление при переключении вкладок
- Split-view внутри `(admin)` layout — админка остаётся вокруг, данные ученика внутри

**Вне scope MVP:**
- Полная имитация сессии ученика (без перелогина)
- Возможность совершать действия от имени ученика (выдача XP, выполнение квестов и т.д.)
- Экраны достижений, магазина, лидерборда, сезонов, босса
- Аудит-лог просмотров (один админ — не нужен)
- Уведомление ученика о просмотре (скрытый режим)

## Требования

### Функциональные
1. Админ может из таблицы `/admin/users` открыть God Mode для любого ученика одним кликом
2. Внутри God Mode отображается снимок данных ученика: герой, статы, инвентарь, активные/завершённые квесты, разблокированные ачивки
3. Админ может переключаться между вкладками Герой / Инвентарь / Квесты
4. Админ может открыть детали артефакта или квеста
5. Кнопка «Обновить» перезагружает все данные
6. При переключении вкладок данные автоматически свежие
7. Все мутирующие кнопки (если такие случайно попадают в dumb-компоненты) показывают toast «Режим просмотра — действия отключены»
8. Закрытие через крестик возвращает в `/admin/users`

### Нефункциональные
1. **Безопасность:** доступ только для `role = 'admin'`. Двойная проверка — в middleware и в API handler.
2. **RLS:** обход через `SUPABASE_SERVICE_ROLE_KEY` строго в одном новом API endpoint. Ни клиент, ни другие маршруты ключом не пользуются.
3. **Нет риска для прод-хуков:** существующие `useHero`, `useArtifacts`, `useHeroStore` ученика не трогаем — они продолжают работать по RLS от имени текущего пользователя.
4. **Изоляция:** новый код живёт в своих файлах, не меняет бизнес-логику ученика.

## Архитектура

### Подход
Approach 1 — **серверный API + dumb-компоненты через props.**

Вместо того чтобы добавлять «режим» в существующие хуки ученика, делаем:
1. Новый серверный API endpoint загружает все данные ученика через SERVICE_ROLE_KEY.
2. Новая страница админки делает fetch на этот endpoint и получает JSON-снимок.
3. Существующие UI-компоненты ученика (или их dumb-обёртки) принимают данные через props и рендерят.

Это даёт нам строгое разделение:
- Student route — как раньше, через хуки и RLS.
- Admin God Mode — через один API endpoint и props.

### Ключевые компоненты

**1. API endpoint**

`GET /api/admin/impersonate/[userId]/route.ts`

- Проверяет: текущий пользователь есть и его `role = 'admin'` (читает из `user_profiles` через обычного клиента)
- Если проверка прошла — создаёт Supabase-клиент с `SUPABASE_SERVICE_ROLE_KEY`
- Загружает параллельно: `heroes`, `hero_stats`, `hero_artifacts` (+ join с `artifacts`), активные и завершённые `quest_attempts` (+ join с `quests`), `achievements_unlocked`
- Возвращает JSON: `{ hero, stats, artifacts, quests: { active, completed }, achievements }`
- Любая ошибка авторизации → `401/403`, никакого fallback'а с пустыми данными

**2. Страница God Mode**

`src/app/(admin)/admin/users/[id]/view/page.tsx`

- Server component или client — client проще, т.к. нужны вкладки и refresh
- При маунте делает `fetch('/api/admin/impersonate/[id]')`
- Хранит снимок в локальном state
- Показывает header: имя/email ученика + кнопки «Обновить» и «✕ Закрыть»
- Внизу — tabs: Герой / Инвентарь / Квесты
- При переключении вкладки перезагружает данные (auto-refresh on tab switch)
- Оборачивает контент в `<ReadOnlyGuard>` (context provider, помечающий всё вложенное как read-only)

**3. Dumb presentational компоненты**

Там, где существующие экраны ученика сильно завязаны на хуки (`useHero`, `useArtifacts` и т.д.), мы выделяем чистую презентационную часть, принимающую данные через props:

- `HeroViewPresentational` — принимает `{ hero, stats }`
- `InventoryViewPresentational` — принимает `{ artifacts }`
- `QuestsViewPresentational` — принимает `{ active, completed }`

Существующие «умные» обёртки в `(student)` остаются и продолжают использовать хуки — они не меняются в поведении, только рефакторятся так, чтобы внутри вызывать dumb-компоненты с данными из хуков.

**4. ReadOnlyGuard**

`src/components/admin/ReadOnlyGuard.tsx`

- React context `{ readOnly: boolean }`
- Хелпер-hook `useReadOnly()`
- Для мутирующих кнопок (если они попадают в dumb-компоненты) — утилита `guardedOnClick(fn)`: если `readOnly` — показывает toast «Режим просмотра — действия отключены», иначе вызывает `fn`

В MVP dumb-компоненты в основном read-only, но guard гарантирует безопасность если что-то проскочит.

**5. Entry point**

В `src/app/(admin)/admin/users/page.tsx` в строку каждого пользователя добавляется кнопка **«👁 Посмотреть»**, ведущая на `/admin/users/[id]/view`.

### Data flow

```
Admin клик на «👁 Посмотреть»
      ↓
/admin/users/[id]/view (client page)
      ↓ fetch
/api/admin/impersonate/[userId]
      ↓ auth check (role=admin)
      ↓ create service-role client
      ↓ parallel queries (hero, stats, artifacts, quests, achievements)
      ↓
JSON snapshot
      ↓
page state → ReadOnlyGuard context → Presentational components → UI
```

### Обработка ошибок

- **Не авторизован / не админ:** API возвращает 401/403, страница показывает «Доступ запрещён» и кнопку «Назад»
- **Ученик не найден:** API возвращает 404, страница показывает «Пользователь не найден»
- **Ошибка загрузки данных:** страница показывает сообщение об ошибке + кнопку «Повторить»
- **Частичные данные (например, у ученика нет героя):** показываем пустое состояние на соответствующей вкладке

### Тестирование

- **Unit:** API handler — проверка auth, возврат правильного JSON, обработка ошибок
- **Unit:** ReadOnlyGuard — toast вызывается, функция не вызывается в read-only
- **Integration:** страница God Mode рендерит данные из mock API, вкладки переключаются, refresh работает
- **Manual smoke:** открыть God Mode для реального тестового ученика, пройти по вкладкам, проверить что клики по мутирующим кнопкам показывают toast

## Безопасность — детали

1. **Middleware** уже защищает `/admin/**` — non-admin получит редирект на логин
2. **API handler** дополнительно проверяет `role = 'admin'` — даже если middleware обойдут, сервер не вернёт данные
3. **SERVICE_ROLE_KEY** живёт только в одном файле (`/api/admin/impersonate/[userId]/route.ts`). Ни один клиентский компонент его не видит.
4. **Нет записей** — endpoint `GET`-only, service-role используется только для чтения
5. **URL не sensitive** — `/admin/users/[id]/view` всё равно защищён middleware
